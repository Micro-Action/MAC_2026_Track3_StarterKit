This repository provides a lightweight baseline runner for Track 3 of the 3rd Micro-Action Analysis Grand Challenge [(MAC 2026)](https://sites.google.com/view/micro-action). 

Track 3 evaluates existing Multimodal Large Language Models (MLLMs) on Fine-grained Micro-Action Understanding (FAMU), using data from our released MA-Bench. To make the competition easy to participate in, this track is formulated as a prompt engineering task: participants design prompts for a fixed MLLMs using methods such as chain-of-thought prompting or in-context learning, **WITHOUT any model fine-tuning**. The runner feeds sampled video frames and a question into an OpenAI-compatible multimodal chat completion endpoint and records the model predictions in a JSON file.

FMAU evaluates fine-grained micro-action understanding from short video clips. Tasks cover body-part movement understanding, temporal order, spatial relations, action descriptions, and coarse or fine micro-action labels. More details are provided in <a href="https://ma-bench.github.io/" target="_blank"><img src="https://img.shields.io/badge/Project-Page-blue?logo=google-chrome&logoColor=white"></a>.

## Guidelines

Participants design an inference-time prompt engineering or in-context learning
(ICL) strategy that enables a fixed, frozen MLLM to answer questions about short
micro-action videos.

| Rule | Requirement |
| --- | --- |
| Goal | Maximize prediction accuracy while keeping answers in the required compact format. |
| Allowed | Zero-shot prompting, few-shot prompting, prompt engineering, frame sampling, and decoding strategies at inference time. |
| Not allowed | Model training, fine-tuning, or using benchmark ground-truth answers. |
| Required output | A JSON file with `prompts` and `answers`; `prompts` must record the final prompts actually used for inference. |

> [!IMPORTANT]
> This is a prompt engineering track. The submitted `prompts` field is part of
> the required submission and should match the prompts used to produce the
> predictions.

## Dataset Format

Each dataset row is expected to contain:

| Field | Description |
| --- | --- |
| `video` | Video id used to locate the clip. |
| `task` | Task name used to select the system prompt from `prompts.py`. |
| `qid` | Unique question id. |
| `question` | Question sent to the model with sampled video frames. |
| `answer` | Ground-truth answer. Present only in the demo split. |

The current runner resolves each clip as:

```text
<VIDEO_DIR>/<video>.mp4
```

`run.py` appends `.mp4` unconditionally. If your JSON `video` value already
contains `.mp4`, either adjust the filename/data field or update
`build_messages()` before running inference.

## Task Types

The current `prompts.py` contains system prompt entries for these task names:

| Task   | Expected answer type | Description |
|--------| --- | --- |
| `CMAC` | Multiple choice, `A`-`G` | Coarse-grained micro-action classification. |
| `FMAC` | Multiple choice, `A`-`E` | Fine-grained micro-action classification. |
| `MAS`  | `Yes` or `No` | Temporal order judgment between two action events. |
| `SAD`  | `Yes` or `No` | Single body-part action description judgment. |
| `MAD`  | `Yes` or `No` | Multi-body-part action description judgment. |
| `PPR`  | `Yes` or `No` | Pairwise spatial or positional relation judgment. |
| `MADU` | Free text | Detailed body-part movement description. |
| `MARE` | Free text | Reasoning for coarse and fine micro-action labels. |

Fill the prompt strings in `prompts.py` before running final inference. The
exported `prompts` field is built directly from the `prompts` dictionary in
that file.

## Environment

Install the Python dependencies:

```bash
pip install openai opencv-python tqdm
```

The runner uses the OpenAI Python SDK. For `LLM_BACKEND=vllm` or `api`, it
connects to `OPENAI_BASE_URL`. For `LLM_BACKEND=openai`, it uses the official
OpenAI API endpoint.

If you use vLLM locally, start an OpenAI-compatible server separately. Example:

```bash
python -m vllm.entrypoints.openai.api_server \
  --model Qwen/Qwen3.5-9B \
  --port 8000 \
  --trust-remote-code
```


`run.py` saves the output JSON after each sample. Interrupted runs can resume
from an existing output file.

## Output Format

The current code exports one JSON object with exactly two top-level fields:
`prompts` and `answers`.

```json
{
  "prompts": [
    {
      "task": "CMAC",
      "system_prompt": "Full system prompt used for CMAC."
    },
    {
      "task": "FMAC",
      "system_prompt": "Full system prompt used for FMAC."
    }
  ],
  "answers": [
    {
      "qid": "MAS_1599",
      "pred_answer": "No"
    },
    {
      "qid": "CMAC_0219",
      "pred_answer": "E"
    }
  ]
}
```

`prompts` is part of the required submission for this prompt engineering track.
It contains one record for every entry in `prompts.py`:

| Field | Description |
| --- | --- |
| `task` | Prompt key from the `prompts` dictionary. |
| `system_prompt` | Full system prompt text used for that task. |

`answers` contains one record for every processed dataset row:

| Field | Description |
| --- | --- |
| `qid` | Question id copied from the dataset row. |
| `pred_answer` | Model prediction after removing a leading `<think>...</think>` block when present. |

The current code does not export `metadata`, `task`, `video`, `question`,
`gt_answer`, `reasoning`, `raw_output`, or `prompt_id`. If inference fails for a
row, `run.py` writes that row with an empty `pred_answer` and prints the error
through `tqdm.write()`.

When resuming, `run.py` can read both the current object format and an older
list-of-results format. It keeps only `qid` and `pred_answer` from existing
records.

## Answer Requirements

Use compact answers for structured tasks:

| Task | Required answer format |
| --- | --- |
| `CMAC` | One option letter from `A` to `G`. |
| `FMAC` | One option letter from `A` to `E`. |
| `MAS`, `SAD`, `MAD`, `PPR` | Exactly `Yes` or `No`. |
| `MADU`, `MARE` | Plain natural-language text. |

The model output is not post-processed into these formats beyond removing
`<think>...</think>`. Put answer-format constraints in `prompts.py` so the model
returns the expected compact answer.

## Checklist

Before using the exported JSON:

- Confirm the file is valid UTF-8 JSON.
- Confirm the top-level fields are `prompts` and `answers`.
- Confirm `prompts` contains the final designed system prompts actually used
  during inference.
- Confirm every expected benchmark `qid` appears once in `answers`.
- Confirm `pred_answer` follows the required format for each task.
- Confirm there are no empty `pred_answer` values caused by runtime errors.

## Notes

- Use `mabench_data_sample_100.json` for local pipeline checks.
- Use `mabench_data.json` for benchmark predictions.
- `run.py` samples frames with OpenCV, encodes them as JPEG data URLs, and sends
  them in one multimodal chat request with the question.
- Make sure the selected model and backend support image inputs through an
  OpenAI-compatible chat completion interface.
